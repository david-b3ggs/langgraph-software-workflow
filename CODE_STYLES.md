# CODE_STYLES.md

## File & Module Layout

```
src/
  └── nodes/
        └── dev_loop/       — Core development loop graph nodes, agent logic, and co-located tests
scripts/                     — Utility and runner scripts
tests/                       — Top-level test suite (additional test root)
```

- **Source code** lives under `src/`. Node and agent implementations are organized by functional area under `src/nodes/`.
- **Graph nodes** for the dev loop are grouped in `src/nodes/dev_loop/`.
- **Scripts** (runners, utilities) go in `scripts/`.
- **New modules** representing graph nodes or agent logic belong under `src/nodes/`, in a subdirectory named for their functional area (e.g., `src/nodes/ingestion/`).
- **Shared state schemas and configuration** belong at the `src/` level or in a dedicated `src/models/` or `src/config/` module if one is created.
- **Context markdown files** (`PROJECT.md`, `CODE_STYLES.md`, `BRAND_STYLES.md`, `TESTING.md`) live at the repository root. They are generated by the ingestion pipeline and consumed by all downstream agents.
- **Do not** place application source files in `scripts/` or `tests/`. Keep `scripts/` for entry-point runners and one-off utilities only.

## Naming Conventions

| Element | Convention | Example |
|---|---|---|
| Files and directories | `snake_case` | `dev_loop/`, `test_loop.py` |
| Python modules | `snake_case` | `context_assembly.py` |
| Classes and Pydantic models | `PascalCase` | `DevLoopState`, `AppSettings` |
| Functions and methods (sync and async) | `snake_case` | `async def run_planner(...)` |
| Constants | `UPPER_SNAKE_CASE` | `ANTHROPIC_MODEL`, `MAX_RETRIES` |
| Test files | Prefixed with `test_` | `test_loop.py` |
| Test functions | Prefixed with `test_` | `async def test_plan_generates_steps():` |
| Environment variables | `UPPER_SNAKE_CASE` | `ANTHROPIC_API_KEY` |

- No special prefix or suffix for async functions — use the same `snake_case` as sync functions.
- Pydantic settings classes follow `PascalCase` like all other classes.

## Async Patterns

- **Async-first.** Graph nodes and agent functions are implemented as `async def`. Prefer async over sync for all I/O-bound operations.
- **Do not create or manage your own event loop.** LangGraph's execution runtime owns the event loop. Node functions must be plain `async def` coroutines — never call `asyncio.run()`, `loop.run_until_complete()`, or similar inside a node.
- **Use async libraries for all I/O:**
  - HTTP requests: `httpx.AsyncClient`
  - SQLite (checkpointing): `aiosqlite`
  - File I/O: use `asyncio` file helpers or standard sync file I/O only when the operation is trivially fast (e.g., reading a small config file).
- **Avoid blocking calls** inside async functions. If you must call a blocking library, offload it with `asyncio.to_thread()`.
- When invoking LangChain runnables or chains from within an async node, use their async interface (e.g., `await chain.ainvoke(...)` rather than `chain.invoke(...)`).

## Error Handling

- **Let exceptions propagate** to the graph runtime by default. LangGraph's checkpointing and retry logic handles recovery at the node-transition level.
- **Use explicit try/except** only when the node has a meaningful recovery action (e.g., retrying an API call with backoff, falling back to a cached result, or escalating to `user_elicitation`).
- **Do not silently swallow exceptions.** If catching an exception, log it before re-raising or handling it.
- **Logging:** Use Python's standard `logging` module. Obtain a module-level logger:
  ```python
  import logging
  logger = logging.getLogger(__name__)
  ```
- **Retry/escalation pattern:** The dev loop graph has built-in retry edges for code review and test failures. After repeated failures, the graph routes to `user_elicitation`. Individual nodes should not implement their own retry loops for graph-level concerns — rely on the graph edges instead.
- **Validation errors:** Use Pydantic's validation for state schemas. Invalid state should raise `ValidationError` immediately rather than being patched up silently.

## Imports & Dependencies

### Import ordering

Follow standard Python convention (PEP 8), enforced in this order with blank lines between groups:

```python
# 1. Standard library
import asyncio
import logging
from pathlib import Path

# 2. Third-party
from langchain_anthropic import ChatAnthropic
from langgraph.graph import StateGraph
from pydantic import BaseModel

# 3. Local / project
from src.nodes.dev_loop.planner import run_planner
```

### Dependency rules

- All runtime dependencies are declared in `requirements.txt` at the repo root with minimum version pins (e.g., `langgraph>=0.2.0`).
- **To add a new dependency:** append it to `requirements.txt` with a `>=` minimum version pin. Do not use exact pins unless there is a known incompatibility.
- **Do not add dependencies for functionality already covered** by the existing stack (e.g., don't add `requests` — use `httpx`; don't add `SQLAlchemy` for checkpointing — use `aiosqlite` via `langgraph-checkpoint-sqlite`).
- **Environment configuration** is loaded from a `.env` file at the project root via `python-dotenv`. Access settings through Pydantic settings classes, not by calling `os.getenv()` directly in business logic.

### Key environment variables

| Variable | Required | Default | Purpose |
|---|---|---|---|

## Testing Approach

### Where tests live

| Location | Purpose |
|---|---|
| `src/nodes/dev_loop/test_loop.py` | Co-located unit/integration tests for dev loop graph logic |
| `tests/` | Top-level test directory (additional test root for cross-cutting or higher-level tests) |

- **Co-locate tests** with their source module when the tests are tightly coupled to a single module (e.g., `src/nodes/dev_loop/test_loop.py` tests `src/nodes/dev_loop/`).
- **Use `tests/`** for tests that span multiple modules or test the system at a higher integration level.
- All test files must follow the `test_*.py` naming convention for pytest discovery.

### Running tests

```bash
# Full suite, verbose, with async support
pytest -v --asyncio-mode=auto

# With coverage
pytest --cov=src --cov-report=term-missing
```

### Writing tests

- Use `pytest` as the test runner. Do not use `unittest.TestCase` unless wrapping legacy code.
- Async test functions are supported via `pytest-asyncio`. Declare them as `async def test_...():` — with `--asyncio-mode=auto`, no decorator is needed.
- Keep tests focused: one behavior per test function.
- Use fixtures for shared setup (e.g., constructing a test state object, mocking LLM responses).
- Mock external services (Anthropic API, LangSmith) in unit tests. Do not make real API calls in the default test suite.
